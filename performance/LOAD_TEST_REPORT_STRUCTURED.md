## Тест 1: Поиск аудиторий (GET /api/v1/rooms)

* **Описание теста:** Тестирование эндпоинта поиска аудиторий с фильтрацией по университету, корпусу и вместимости. Сценарий `burst_read`: ramping нагрузка от 44 до 440 RPS, длительность 2 минуты, максимум 1000 виртуальных пользователей (VUs). Дополнительно сценарий `mixed_load`: 60 RPS (85% read / 15% write), длительность 5 минут, максимум 200 VUs.

* **Ожидаемый результат:** 
  - Целевой RPS: 440 req/s (burst_read) + 51 req/s (mixed_load read portion)
  - SLA по времени ответа: p95 < 200ms
  - Допустимый процент ошибок: 0%
  - HTTP 500 errors: 0% (threshold: <0.05%)

* **Итоговый результат:**
  - Фактически достигнутый RPS: ~47 req/s (из общих 55.59 req/s, учитывая 85% read в mixed_load)
  - Реальный p95 Latency: 23.39s (для всех запросов)
  - Процент ошибок: 0% (search ok: 100% success rate)
  - HTTP 500 errors: 0.00% ✅

* **Вывод:** Тест **частично пройден**. HTTP 500 ошибки полностью устранены (было 88 ошибок в первом прогоне). Endpoint стабильно работает без ошибок, но производительность критически низкая: достигнуто только ~10% от целевого RPS (47 vs 440). p95 latency 23.39s в 117 раз превышает целевой SLA (<200ms). Основные причины: thread pool exhaustion (Tomcat default 200 threads), memory pressure (71% usage), отсутствие кэширования.

## Тест 2: Проверка доступности аудитории (GET /api/v1/rooms/{id}/availability)

* **Описание теста:** Тестирование эндпоинта проверки доступности конкретной аудитории на заданную дату. Выполняется в рамках сценариев `burst_read` и `mixed_load`. Эндпоинт делает gRPC-вызов к Availability Engine (C++) для расчета свободных временных окон на основе расписания.

* **Ожидаемый результат:**
  - Целевой RPS: часть от общего read traffic (~200 req/s)
  - SLA по времени ответа: p95 < 300ms (допускается выше из-за gRPC overhead)
  - Допустимый процент ошибок: <5%
  - Availability endpoint success rate: >95%

* **Итоговый результат:**
  - Фактически достигнутый RPS: ~8 req/s (оценочно, из общего read traffic)
  - Реальный p95 Latency: 23.39s (общая метрика для всех запросов)
  - Процент ошибок: 0% (avail ok: 100% success rate) ✅
  - В первом прогоне было 51% failure rate, после исправлений - 100% success

* **Вывод:** Тест **пройден по стабильности**, но **не пройден по производительности**. После исправления foreign key violations endpoint работает стабильно (100% success rate vs 49% в первом прогоне). Однако производительность критически низкая: RPS ~8 vs целевые 200, latency 23.39s vs целевые 300ms. Узкое место: отсутствие кэширования результатов Availability Engine - каждый запрос = gRPC call. Рекомендация: Redis cache с TTL 30-60s.

## Тест 3: Создание бронирования (POST /api/v1/bookings)

* **Описание теста:** Тестирование эндпоинта создания бронирования аудитории. Два сценария: `race_write` (15 constant VUs, 30 секунд) для проверки race conditions и `mixed_load` (15% write traffic, 60 RPS total). Тест намеренно создает конфликтующие бронирования для проверки работы constraint `no_overlap`.

* **Ожидаемый результат:**
  - Целевой RPS: ~9 req/s (15% от 60 RPS в mixed_load)
  - SLA по времени ответа: p95 < 500ms (write операции допускают больший latency)
  - Допустимый процент ошибок: <1% (исключая ожидаемые HTTP 409 Conflict от race conditions)
  - HTTP 500 errors: 0%

* **Итоговый результат:**
  - Фактически достигнутый RPS: ~0.6 req/s (1,412 total bookings за 7m29s)
  - Реальный p95 Latency: 23.39s (общая метрика)
  - Процент ошибок: 10% (write ok: 90% success rate - 1,280 ✓ / 132 ✗)
  - HTTP 500 errors: 0.00% ✅
  - Тип ошибок: HTTP 409 Conflict (constraint `no_overlap` violation)

* **Вывод:** Тест **пройден по корректности**, но **не пройден по производительности**. HTTP 500 ошибки полностью устранены (было 88 foreign key violations в первом прогоне). Constraint `no_overlap` корректно блокирует конфликтующие бронирования (10% failures - это **ожидаемое поведение** для race condition теста). Однако производительность критически низкая: RPS 0.6 vs целевые 9, latency 23.39s vs целевые 500ms. Требуется обработка HTTP 409 на frontend с retry logic и exponential backoff.

  **Выявленные узкие места (Bottlenecks):**

  1. **Thread Pool Exhaustion (критично)**
     - Tomcat default: 200 threads
     - При нагрузке >100 RPS все threads заняты, новые запросы блокируются
     - Рекомендация: `server.tomcat.threads.max=500`, `server.tomcat.threads.min-spare=50`

  2. **Memory Pressure (критично)**
     - booking-api: 710.6 MiB / 1 GiB (71% usage)
     - Близко к лимиту, возможны GC паузы
     - Рекомендация: Увеличить memory limit до 2GB

  3. **Отсутствие кэширования Availability Engine (критично)**
     - Каждый запрос = gRPC call к C++ сервису
     - Нет кэширования результатов расчета доступности
     - Рекомендация: Redis cache с TTL 30-60s для результатов availability

  4. **Отсутствие индексов на часто используемых полях (высокий приоритет)**
     - Отсутствует индекс на `rooms(capacity)` для фильтрации
     - Рекомендация: `CREATE INDEX idx_rooms_capacity ON rooms(capacity);`

  5. **Отсутствие Circuit Breaker для Availability Engine (высокий приоритет)**
     - При деградации C++ сервиса вся система блокируется
     - Рекомендация: Resilience4j Circuit Breaker с fallback
