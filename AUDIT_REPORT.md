# Аудит и E2E-тестирование проекта "Poly Booking"

## 1. Состояние кода и документации
*   **gRPC Заглушки:** В `availabilityengine` (C++) методы `ComputeIntervals` и `OcupiedIntervals` являются пустыми заглушками, хотя `booking_API` (Java) вызывает `ComputeIntervals` при поиске комнат. Это приводит к тому, что поиск с фильтрами по времени всегда возвращает пустой список.
*   **Дублирование логики:** `RoomService.java` напрямую обращается к PostgreSQL для построения таймлайна (`getRoomSchedule`), минуя `Availability Engine`, что нарушает архитектурный принцип централизации вычислений доступности.
*   **Рассинхрон миграций:** Обнаружены две директории с миграциями (`infrastructure` и `booking_API`). Docker использует только инфраструктурные, в которых отсутствуют `INSERT` для пользователей и комнат, описанные в документации.
*   **Мертвый код в NATS:** Метод `publishScheduleRefreshed` в C++ клиенте NATS реализован, но нигде не вызывается. События об обновлении расписания не генерируются.
*   **Фейковый Redis:** В `ruz-importer` соединение с Redis полностью имитируется (stubbed). Код устанавливает `is_connected = true` без реальной сетевой активности.

## 2. Отчет о запуске контейнеров
*   **Статус:** Все 13 контейнеров успешно поднялись и прошли Healthchecks.
*   **Сетевые конфликты:** Обнаружены конфликты портов `5432`, `6379`, `4222` с локальными сервисами хоста. Решено путем переопределения в `.env` (5433, 6380, 4223).
*   **База данных:** Миграции Flyway (14 шагов) применены успешно.

## 3. Результаты базового тестирования (API Smoke Tests)
*   `POST /api/v1/auth/login`: **Успех (200 OK)**. Использован пользователь `admin@arch.local` (создается через `@PostConstruct` в Java).
*   `GET /api/v1/rooms`: **Успех (200 OK)**. Возвращает список после ручной синхронизации метаданных. До синхронизации возвращал `[]`.
*   `GET /rooms/{id}/availability`: **Ограниченно (200 OK, но пустой результат)**. Возвращает `[]` из-за заглушки в C++.
*   `POST /api/v1/bookings`: **Успех (201 Created)**. Бронирование создается в статусе `pending`.
*   `GET /api/v1/bookings/{id}`: **Успех (200 OK)**. Через 1-2 секунды статус меняется на `confirmed`, что подтверждает работоспособность связки **Saga -> gRPC (Validate) -> PostgreSQL**.

## 4. Глубокий анализ RUZ Importer (Критический узел)
*   **Статус запуска:** OK. Подключение к PostgreSQL и NATS подтверждено.
*   **Парсинг:** Работает. За 1 минуту работы было импортировано более 1000 записей из RUZ API (`spbgu`, `leti`, `spbptu`).
*   **Состояние БД:** Записи в `schedules_import` появляются успешно.
*   **Авто-наполнение:** **FAIL**. Миграция `V000013`, отвечающая за создание комнат из импорта, запускается *до* того, как импортер скачает данные. Комнаты и здания не создаются автоматически при холодном старте.
*   **Трансляция событий:** **FAIL**. События в NATS не отправляются (отсутствует вызов в коде).

## 5. Анализ общих логов (Критические ошибки)
*   **Сервис `ruz-importer`:**
    *   Ошибка: Ложное подтверждение соединения с Redis.
    *   Лог: `[INFO ] Attempting to connect to Redis server at localhost:6379`, `[INFO ] Redis connection established successfully.` (даже если хост неверный).
*   **Сервис `booking-api`:**
    *   Предупреждение: Неоднозначность репозиториев (Spring Data JPA vs Redis).
    *   Лог: `Spring Data Redis - Could not safely identify store assignment for repository candidate interface...`
